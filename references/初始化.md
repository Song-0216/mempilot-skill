# 初始化

## 目标

首次轻量分析项目，自动生成中文项目记忆和当前 Agent 原生入口。

## 扫描步骤、产出与停止条件

| 步骤 | 扫描对象 | 本步必须产出 | 何时停止 |
|---|---|---|---|
| 1 | 根目录和主代码目录的一到两级结构 | 项目类型、主代码位置、候选功能域 | 已识别项目类型和主代码边界 |
| 2 | README 与已有文档 | 一句话项目定位、目标用户、运行方式、已知约束 | 能清楚描述项目做什么 |
| 3 | 依赖清单、构建和运行配置 | 框架、关键库、运行环境、构建与部署方式 | 已识别主要技术栈和启动方式 |
| 4 | 用户可达入口或程序入口 | 页面路由、CLI 命令、API 路由、扩展入口、桌面窗口或公开导出列表 | 已列出主要交互入口 |
| 5 | 数据模型、Schema 和迁移 | 核心实体、关联、存储与迁移方式 | 能判断主要数据边界 |
| 6 | API、服务层、状态管理入口 | 模块间调用、前后端数据流、外部服务边界 | 能说明主要数据如何流动 |
| 7 | 测试结构 | 已覆盖领域、验证方式、高风险空白 | 能说明项目如何验证 |
| 8 | Git 或会话变更信息 | 当前阶段、近期热点、正在开发的模块 | 能确认当前重点或明确写待确认 |
| 9 | 已有 Agent 规则文件 | 需要保留的工具专属规则与入口位置 | 已确认如何合并入口 |

“用户可达入口或程序入口”按项目类型理解：Web 看 URL/菜单；CLI 看命令和参数解析；浏览器扩展看 `manifest.json`、background、content scripts、popup；API 服务看路由注册；桌面应用看窗口、菜单和 IPC；库项目看公开导出。

当已经能可靠回答以下问题时，停止扩大扫描：项目是什么、用户如何进入、核心模块有哪些、模块如何交互、数据存在哪里、当前开发重点是什么、哪些信息仍待确认。模块边界仍不清楚时，才深入代表性文件；禁止全文读取整个仓库。

## 模块识别

模块应满足多个条件：独立职责、长期反复开发、独立数据流或约束、未来值得单独读取。高度重叠的候选合并。

普通工具函数、单页面、单组件、类型文件、样式和临时实验通常不建模块。

## 创建

1. 创建固定骨架。
2. 根据真实项目创建 `模块/*.md`，使用 `assets/模块模板.md` 的填写指导。
3. 创建 `当前状态.md`、`项目索引.md`、`调试记忆.md`。
4. 确有复杂接口、字段、迁移或第三方依赖时才创建 `接口契约.md`。
5. 根据当前 Agent 读取 `references/Agent入口.md` 并生成入口。
6. 正式模块文件不得保留空章节；没有可靠内容时写“待确认”，不得编造。

## 初始化真实值

初始化结束前必须：

- 取得真实当前日期，不得保留日期占位符。
- Git 仓库且已有提交时执行 `git rev-parse HEAD`。
- 非 Git 项目或尚无提交时写：`无可用提交（非 Git 项目或尚无提交）`。
- 禁止保留“未执行”“待记录”“<commit>”等占位内容。

在 `项目索引.md` 中写入：初始化日期、上次快速维护日期（注明初始化检查）、深度维护为“尚未需要”、Git 基线、健康状态和具体下次建议。

## 结束

写入模块语义路标，每个模块 3～8 个来自真实业务、路由、接口、字段或用户术语的关键词。初始化后冻结结构，普通开发不重新规划全部目录。
