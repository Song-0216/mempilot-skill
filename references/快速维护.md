# 快速维护与健康报告

## 范围

### 有 Git 时

优先检查上次维护 Git 基线到当前 HEAD 的变化，不全量重读项目。

### 没有 Git 时

优先检查：上次维护后用户确认的变更范围、本次会话修改记录、当前任务涉及路径、最近维护日期和最近检查模块。无法确认范围时，明确报告低置信并请求用户指定近期主要改动。

通用检查：当前状态是否过期、会话交接是否矛盾、语义路标是否有效、最近变化是否漏写、文件是否膨胀、重复事实、过期快照、Agent 入口和维护日期。

## 最小腐烂检测

快速维护不得全量验证所有记忆。优先选择：当前任务相关模块、最近变化模块、距离上次检查最久的一个模块，默认共 1～3 个。

每个模块：

- 从“相关代码位置”抽查 3～5 个路径是否存在。
- 至少覆盖一个入口文件、一个核心实现文件，以及一个测试或数据文件（项目存在时）。
- 最多搜索 3 个关键 API、字段或配置项是否仍存在。
- 检查 3～8 个路标词是否仍能命中当前代码或用户入口。
- 失效项标记为：路径已失效、可能重命名、模块已迁移或待确认。

## 轻量语义漂移抽查

仅检查已选模块，不扩大到全库。每个模块最多抽查 3 条高影响陈述：

1. `当前有效结构` 中一条架构或组成陈述；
2. `关键流程` 中一条入口到结果的流程陈述；
3. `已确认约束` 中一条会影响实现的规则。

对每条陈述定位最小充分证据：入口、核心实现、配置、数据模型或测试中的代表性片段。结果只能标记为：

- `已验证`：当前代码明确支持；
- `疑似漂移`：代码出现相反路径、新分支或职责变化；
- `证据不足`：无法用有限读取可靠确认。

只有 `已验证` 的明确变化才可局部更新记忆。`疑似漂移` 和 `证据不足` 必须报告，不能静默改写。单模块发现 2 条及以上疑似漂移，或多个模块同时漂移时，停止扩大检查并提出深度维护。

## 自动可修复

- 明显失效链接和已确认的简单重命名路径
- 当前状态中的已完成旧任务
- 失效或缺失的路标关键词
- 重复短句
- 维护日期和 Git/非 Git 基线信息
- 明显低频的旧 Bug 归档

不能确认是简单重命名还是职责变化时，不自动替换，先报告。

## 先报告再处理

新增、拆分、合并、重命名模块；删除记忆文件；大规模改写；推翻 ADR；大量清理历史。

## 主动维护提议与抑制

若检查命中维护阈值，必须列出信号、影响和建议模式，并询问是否现在执行。用户同意快速维护后可立即执行；建议深度维护时仍需先展示结构变更方案。

对每个提醒生成指纹：`信号类型 + 文件或模块 + 阈值/条件`。在 `项目索引.md` 的维护状态中最多保存 5 条未解决指纹、提醒日期和当时 Git 基线；无 Git 时保存相关文件范围。

同一未变化提醒默认抑制 7 天。以下任一条件允许提前再次提醒：

- Git 基线变化，或非 Git 项目的相关文件再次变化；
- 数值型严重程度比上次增加约 20%；
- 完成版本、迁移或大重构；
- 用户主动询问健康状态或要求检查；
- 上次提醒的问题已经变化为新的信号类型。

问题解决后删除对应指纹，不保留提醒历史正文。

## 健康报告模式

用户只要报告时，禁止修改。输出：问题、证据、严重度、建议动作和健康状态。
